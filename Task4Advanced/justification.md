# Архитектурное обоснование: Событийный подход (EDA) против Legacy (ESB Camel & SQL Server DWH)

Для компании «Будущее 2.0» переход от синхронного интеграционного монолита на базе **Apache Camel** и **MS SQL Server 2008** к асинхронной событийно-ориентированной платформе (**Event-Driven Architecture на базе Apache Kafka**) — критическая необходимость для выживания бизнеса.

Ниже приведено детальное сравнение, демонстрирующее преимущества целевой архитектуры по ключевым метрикам.

### 1. Гибкость и скорость подключения новых направлений (Time-to-Market)
*   **Как было (Camel / DWH):** При покупке банком фармацевтической компании инженерам приходилось переписывать сквозную логику внутри старой шины данных Camel (сложные XML/Java маршруты) и дорабатывать таблицы/хранимые процедуры в MS SQL Server 2008. Любое изменение в одном домене приводило к каскадным ошибкам в других.
*   **Как стало (EDA / Kafka):** Домены полностью изолированы. Домен «Фармацевтика» просто подключается к брокеру сообщений и начинает читать топик `medical-events` (например, событие `MedicationPrescribed` — выписан рецепт), чтобы автоматически планировать отгрузку лекарств. Домену «Фарма» **не требуется доступ в чужие базы данных**. Время интеграции нового бизнеса сокращается с месяцев до дней.

### 2. Масштабируемость при обработке сотен терабайт (Scalability)
*   **Как было (Camel / DWH):** Старая шина Camel работает по синхронной модели (Request-Response). Тяжелые файлы (результаты КТ/МРТ исследований) или лавина банковских транзакций создавали пиковые нагрузки, которые приводили к отказам памяти. Центральное DWH на SQL Server 2008 задыхалось от сотен терабайт, а построение аналитических отчетов занимало часы из-за блокировок таблиц (деградация OLTP-нагрузки клиник).
*   **Как стало (EDA / Kafka):** Kafka спроектирована как распределенный отказоустойчивый лог событий с линейным горизонтальным масштабированием (Partitioning). Брокер сглаживает пиковые нагрузки: если ИИ-сервис временно замедлился из-за наплыва тяжелых КТ-снимков, сообщения накапливаются в Kafka, не роняя клиники. Аналитическое хранилище (ClickHouse) получает данные инкрементально потоком в реальном времени, полностью исключая ночные "batch-окна" и многочасовые ожидания отчетов.

### 3. Отказоустойчивость и автономность доменов (Resilience & Availability)
*   **Как было (Camel / DWH):** Плотная связность (Tight Coupling). Если падал центральный SQL Server или зависала шина Camel, останавливалась работа всей компании: врачи не могли открыть АРМ, ИИ не анализировал снимки, банк не мог провести оплату.
*   **Как стало (EDA / Kafka):** Слабая связность (Loose Coupling). Если в банке проводятся технические работы, клиники продолжают принимать пациентов. События `InvoiceIssued` (выставлен счет) просто накапливаются в Kafka. Как только банковский микросервис возвращается в строй, он вычитывает накопившуюся очередь сообщений и завершает транзакции. Для клиентов и врачей система доступна 24/7.

### Сводная таблица сравнения подходов


| Метрика | Текущее состояние (Legacy монолит) | Целевое состояние (3 года — EDA) |
| :--- | :--- | :--- |
| **Связность компонентов** | Высокая (Синхронные RPC / Общая БД) | Слабая (Асинхронные события) |
| **Единая точка отказа** | Да (SQL Server 2008 / Настройки Camel) | Нет (Распределенный кластер Kafka / Независимые БД доменов) |
| **Скорость аналитики** | Пакетная (Batch), задержка часами | Возле реального времени (Near-Real-Time, < 5 секунд) |
| **Изоляция безопасности** | Нарушена (Данные банка и медкарт лежат вместе) | Абсолютная (В Kafka и OLAP DWH уходят только хэши и бизнес-метрики) |
